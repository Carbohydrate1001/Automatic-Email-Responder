## 2026-03-09

### 新增功能
1. **邮件发送失败识别与重试机制**
   - 新增发送重试配置项：`SEND_RETRY_MAX_ATTEMPTS`、`SEND_RETRY_DELAY_SECONDS`。
   - Graph 发信流程支持自动重试，失败后抛出明确异常。
   - 自动发送与人工发送流程均接入重试逻辑。

2. **失败状态可观测**
   - 增加状态：`send_failed`（发送失败，可重试）。
   - 邮件记录增加字段：`retry_count`、`last_error`，用于追踪失败次数与原因。
   - 列表/详情/统计页同步展示失败状态与数量。

3. **非业务邮件识别与自动忽略**
   - 分类新增：`non_business`（非业务邮件，无需回复）。
   - 通过提示词规则强化 + 规则兜底（如通知/营销/系统消息）降低误分类。
   - 对 `non_business` 邮件不生成回复、不发送，状态置为 `ignored_no_reply`。

4. **仓库忽略规则完善**
   - 新增 `backend/.gitignore`，忽略 `backend/.env`。
   - 已将 `backend/.env` 从 Git 跟踪移除（本地文件保留）。

### Bug 修复
1. **发送故障无识别、无重试**
   - 问题：邮件发送失败后系统无法感知，流程中断且无可追踪信息。
   - 修复：补齐错误捕获、失败状态写入、重试机制与失败详情回传。

2. **非业务内容被高置信度误分为业务并自动回复**
   - 问题：如 OneDrive 通知类邮件被误分到 `order_tracking` 并自动发送回复。
   - 修复：新增 `non_business` 类别及规则过滤，并将该类邮件自动忽略。

## 2026-03-10

### 新增功能
1. **拉取流程稳健性增强（逐封隔离）**
   - `POST /api/fetch` 改为逐封 `try/except`，单封失败不再中断整批处理。
   - 新增返回字段：`failed`、`errors`、`status_breakdown`，便于定位失败邮件。

2. **两阶段分类判定（业务相关性 Gate + 业务类别分类）**
   - 新增业务相关性判定步骤：先判断邮件是否属于业务服务请求，再执行具体类别分类。
   - 对判定为非业务的邮件直接归类 `non_business`，减少“关键词误判为业务”的概率。

3. **自动发送策略收紧**
   - 自动发送增加安全门槛：必须业务相关且满足阈值条件，进一步降低误发风险。
   - 非业务邮件统一标记为 `ignored_no_reply`，不生成、不发送回复。

4. **可观测性指标增强**
   - 增加结构化流水日志（`message_id`、`status`、`category`、`confidence`、`latency_ms`）。
   - 拉取接口返回处理时延指标：`total_elapsed_ms`、`avg_per_email_ms`。
   - 统计接口新增 `non_business_rate`、`send_failure_rate`。

### Bug 修复
1. **批处理中断问题**
   - 问题：任意一封邮件处理异常会导致本次批量拉取整体失败。
   - 修复：改为逐封隔离处理并记录失败明细，保证其余邮件继续处理。

2. **误分类导致误发送风险仍偏高**
   - 问题：部分非业务邮件在关键词干扰下仍可能进入业务分类并触发自动发送。
   - 修复：引入业务相关性 Gate，并收紧自动发送条件，优先将非业务邮件落入 `ignored_no_reply`。

   - 涉及模块：`backend/services/reply_service.py`。

   - 涉及模块：`backend/services/company_info_service.py`、`backend/routes/company_routes.py`、`backend/data/company_products.json`。

4. **文档同步更新**
   - 更新 `non_business` 类别说明与处理流程文案，明确“生成固定说明模板但不自动发送”。
   - 涉及模块：`使用说明.md`。

### Bug 修复

   - 修复：新增 `billing_invoice` 专用模板并接入类别分发逻辑。

## 2026-04-22

### 新增功能

1. **分类 Prompt 完整重构（Few-Shot + 消歧规则）**
   - 新建 `backend/data/few_shot_examples.json`，包含 7 类别各 2 正例 + 1 反例，共 21 条标注样本。
   - `CATEGORY_PROMPT` 改为动态构建函数 `_build_category_prompt()`，模块加载时从 JSON 文件读取并嵌入示例。
   - 新增 **DISAMBIGUATION RULES** 区块（7 条消歧规则），解决 order_tracking vs shipping_time、shipping_exception vs order_cancellation 等边界混淆问题。
   - 重构 `BUSINESS_GATE_PROMPT`，明确列举 false 触发类型（Microsoft 通知、订阅邮件、营销邮件等）。
   - Gate 判断阈值从 0.6 提升到 0.7，减少业务邮件被误判为非业务的概率。
   - 涉及模块：`backend/services/classification_service.py`、`backend/data/few_shot_examples.json`。

2. **回复内容全面改造（消除假数据，基于上下文生成）**
   - 新增辅助方法：`_detect_language()`（CJK字符占比检测）、`_extract_tracking_number()`（多格式运单号提取）、`_extract_mentioned_amount()`（金额提取）、`_summarize_exception_full()`（改进版异常描述提取，优先抽取含关键词句子，最多300字）。
   - **`order_tracking`**：优先从邮件中提取真实订单号/运单号；若无则引导客户提供，不再生成随机订单号或硬编码日期。
   - **`shipping_time`**：ETA 改为从产品数据库动态计算（`received_at + delivery_lead_time_days`），删除硬编码 `”2026/3/16”`。
   - **`billing_invoice`**：引用邮件中提到的真实金额和订单号，删除随机生成的账单号、金额和发票信息。
   - **`shipping_exception`**：使用改进的异常描述提取，提取真实运单号，不再生成随机订单号。
   - **`order_cancellation`**：提取真实订单号，友好提示处理时效。
   - 所有模板：支持中/英双语自动切换（基于邮件来件语言检测），提取客户邮箱前缀作为称呼。
   - 产品匹配增强：支持 `aliases`（JSON 数组别名）和分词模糊匹配（例如 “Sea Freight” 的单词均出现则匹配）。
   - 涉及模块：`backend/services/reply_service.py`。

3. **数据库扩建（新增 4 张表 + 字段扩展）**
   - 新增 `feedback_records` 表：记录每次人工批准/拒绝/编辑操作，包含分类纠正记录（`original_category` vs `corrected_category`）和回复编辑记录。
   - 新增 `few_shot_examples` 表：可随时从 Web 界面或后台追加标注示例，支持从反馈晋升。
   - 新增 `products` 表：将产品信息从 JSON 文件扩展为数据库表，支持 `aliases` 和 `description` 字段。
   - 新增 `customers` 表：记录客户联系历史（联系次数、最后联系时间、偏好语言）。
   - `emails` 表新增字段：`language`、`is_read`、`human_category`（人工纠正分类）、`feedback`。
   - `replies` 表新增字段：`feedback_score`、`edited_text`。
   - 所有扩展通过 `_ensure_column()` 向后兼容，不影响已有数据。
   - 涉及模块：`backend/models/database.py`。

4. **人工反馈回路（操作历史持久化）**
   - `POST /api/emails/<id>/reject`：新增可选请求体字段 `corrected_category`，记录人工认为的正确分类；写入 `feedback_records` 表；更新 `emails.human_category`。
   - `POST /api/emails/<id>/approve`：检测回复是否被人工编辑，写入 `feedback_records`（`action='approved_with_edit'` 或 `'approved'`）；更新 `customers` 表联系计数。
   - 新增端点 `GET /api/feedback/stats`：返回分类纠正统计（哪些类别最常被纠正）和回复编辑率。
   - 涉及模块：`backend/routes/email_routes.py`。

5. **系统稳定性加固**
   - Graph API 发送重试策略改为**指数退避**（1s → 2s → 4s，上限 30s），替代原来的固定间隔。
   - `POST /api/fetch` 新增 `MAX_FETCH_LIMIT` 上限保护（默认 20），防止单次处理过多邮件导致 OpenAI 并发超限，支持环境变量覆盖。
   - 涉及模块：`backend/services/graph_service.py`、`backend/routes/email_routes.py`。

6. **前端优化与修复**
   - 修复 `stores/email.ts` `rejectEmail()` 缺少 try-catch 导致按钮永久禁用的问题。
   - 修复 `stores/email.ts` `approveEmail()` 在 `finally` 中刷新数据（即使发送失败也覆盖错误状态）的问题，改为仅在成功时全量刷新。
   - 修复 `EmailDetailView.vue` `handleReject()` 缺少 `try-finally`，`rejecting` 状态不重置的问题。
   - `sanitizeBody()` 改用 `div.textContent` 替代正则替换，防止潜在 XSS 风险。
   - 搜索框改为 350ms 防抖 `@input`，避免每次按键触发请求。
   - 回复草稿编辑自动保存到 `localStorage`（500ms 防抖），页面刷新后自动恢复；发送成功后自动清除。
   - `rejectEmail` API 支持传递 `corrected_category` 参数（对接后端反馈回路）。
   - 导航栏新增”产品管理”入口，新建 `ProductsView.vue` 产品管理页面（表格展示 + 新增/编辑/删除）。
   - `api/index.ts` 新增 `companyApi`（listProducts / addProduct / upsertProduct / deleteProduct）和 `getFeedbackStats`。
   - 涉及模块：`frontend/src/stores/email.ts`、`frontend/src/views/EmailDetailView.vue`、`frontend/src/views/EmailListView.vue`、`frontend/src/views/ProductsView.vue`（新建）、`frontend/src/api/index.ts`、`frontend/src/router/index.ts`、`frontend/src/components/AppHeader.vue`。

### Bug 修复

1. **回复中使用随机假数据（订单号/金额/ETA 硬编码）**
   - 问题：系统发给客户的回复含随机生成的订单号、固定日期 `2026/3/16`、随机账单金额，与客户真实订单无关联。
   - 修复：重构所有模板方法，改为从邮件正文提取真实信息；无法提取时引导客户提供而非生成假数据。

2. **前端拒绝操作异常后按钮永久禁用**
   - 问题：`rejectEmail()` 无错误处理，API 失败时 `rejecting.value` 不重置。
   - 修复：添加 `try-catch-finally`，确保异常时 `rejecting` 重置，并显示友好错误提示。

3. **审核通过失败时仍刷新邮件状态导致 UI 不一致**
   - 问题：`approveEmail()` 在 `finally` 块中刷新，即使发送失败也覆盖了正确的 `send_failed` 状态显示。
   - 修复：将全量刷新移入 `try` 成功分支，失败时仅刷新当前邮件以展示 `send_failed` 状态。
