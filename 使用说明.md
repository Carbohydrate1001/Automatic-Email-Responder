## 1. 系统概述

本系统面向物流/贸易等场景，通过 **Microsoft Graph API** 读取 Outlook 邮箱中的客户邮件，结合 **OpenAI 兼容 API** 完成分类与处理，主要能力包括：

- **意图分类**：规则关键词辅助 + 业务相关性 Gate + 多类别分类（7 个业务/非业务类别，定义见 `backend/config/categories.yaml`）
- **回复生成**：各意图优先使用**业务模板**（如询价会引用公司产品库 `backend/data/company_products.json`）；未覆盖类别时回退为 **LLM 生成**草稿
- **自动发送决策**：在置信度之外，还经过 **Rubric 自动发送评分** 与 **回复质量校验**；任一环节不通过则进入人工审核
- **合规与隐私**：处理流程中会进行 **PII 检测** 等记录（见数据库扩展字段）
- **人工审核**：未自动发送、发送失败等邮件可在 Web 端审核、编辑后发送
- **产品信息库（JSON）**：维护产品名、单价、MOQ、交付周期等，供询价类模板引用（当前无独立管理页，见下文）

**技术架构**

```
浏览器 (Vue 3 · 默认端口 5173)
        ↕  HTTP + CORS，Cookie Session
Flask API 服务 (默认端口 5005)
    ├── Microsoft Graph API  ←  读/发 Outlook 邮件
    ├── OpenAI 兼容 API  ←  分类、评分、部分草稿
    ├── YAML 配置 (backend/config/)  ←  类别、阈值、策略
    └── SQLite (email_system.db) + 公司 JSON 产品库
```

## 3. Azure AD 应用注册

> 使用 Microsoft 身份平台 OAuth2。个人 Microsoft 账号常用租户 ID：`consumers`（与项目 `backend/.env.example` 一致）。

### 3.1 创建应用注册

1. 访问 [Azure 门户](https://portal.azure.com) → 搜索 **「应用注册」** → **新注册**
2. 名称自填，账户类型常选 **「任何组织目录中的帐户和个人 Microsoft 帐户」**（或按你的租户策略选择）
3. 重定向 URI 选 **Web**，填入（须与 `.env` 中完全一致）：

   ```text
   http://localhost:5005/auth/callback
   ```

4. 记录 **应用程序(客户端) ID** → `AZURE_CLIENT_ID`，**目录(租户) ID** 或个人账号场景用 `consumers` → `AZURE_TENANT_ID`

### 3.2 创建客户端密码

1. **证书和密码** → **新建客户端密码** → 复制 **值** → `AZURE_CLIENT_SECRET`（只显示一次）

### 3.3 API 权限

**API 权限** → **添加权限** → **Microsoft Graph** → **委托权限**，勾选并同意：

- `User.Read`
- `Mail.Read`
- `Mail.ReadWrite`
- `Mail.Send`

若界面提供 **代表管理员同意**，请视组织策略执行。

## 4. 后端配置与启动

### 4.1 安装依赖

```bash
cd backend

python -m venv venv
# Windows
venv\Scripts\activate
# macOS / Linux
# source venv/bin/activate

pip install -r requirements.txt
```

### 4.2 环境变量

```bash
cd backend
cp .env.example .env
```

在 `.env` 中至少配置以下项（示例值请替换为真实值）：

```env
# Flask
FLASK_SECRET_KEY=请使用随机长字符串
FLASK_DEBUG=True
FLASK_PORT=5005

# Azure
AZURE_CLIENT_ID=你的应用客户端 ID
AZURE_CLIENT_SECRET=你的客户端密码
AZURE_TENANT_ID=consumers
AZURE_REDIRECT_URI=http://localhost:5005/auth/callback

# OpenAI 兼容 API
OPENAI_API_KEY=你的 Key
# 可留空则使用 config.py 中的默认 Base URL；若使用官方可写：
# OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4o-mini

# 分类最低置信度（与 YAML 中 global.confidence_threshold 等配合使用）
CONFIDENCE_THRESHOLD=0.75

# 发信失败重试（与 graph_service 等配合）
SEND_RETRY_MAX_ATTEMPTS=3
SEND_RETRY_DELAY_SECONDS=1.0
```

`.env` 已被忽略提交，**勿将密钥推送到仓库**。

### 4.3 启动

```bash
# 在 backend 目录、虚拟环境已激活
python app.py
```

成功时控制台会打印本机 `5005` 服务地址。浏览器访问根路径应得到 JSON：`status: running`。

## 5. 前端配置与启动

```bash
cd frontend
npm install
npm run dev
```

访问：**http://localhost:5173**

当前前端的 HTTP 客户端在 `frontend/src/api/index.ts` 中直接请求 `http://localhost:5005`（`withCredentials: true` 携带会话 Cookie），后端已对该来源启用 CORS。`vite.config.ts` 中虽配置了 `/api`、`/auth` 代理，**默认前端实现未走代理**；若需同域代理，可将 `baseURL` 改为相对路径并自行验证流程。

**构建（可选）**

```bash
npm run build
# 产物在 frontend/dist/
```

## 6. 演示前准备与推荐演示流程

按下列顺序准备，可完整走通「拉取 → 分类 → 自动/待审 → 看板」演示。

### 6.1 演示前检查清单

| 项 | 说明 |
|----|------|
| 后端 | `5005` 可访问，无启动报错；`.env` 中 Azure 与 OpenAI 已填 |
| Azure | 重定向 URI 与 `AZURE_REDIRECT_URI` 一字不差；已授予 Graph 邮件与发信权限 |
| 邮箱 | 用 **同一微软账号** 登录，且收件箱里**有几封**可用来演示的邮件（**新**邮件在首次拉取时才会进库，已按 `message_id` 去重） |
| 产品库（可选） | 若要突出「询价+报价单」，可事先编辑 `backend/data/company_products.json`，**保存后需重启或确保进程读到的文件已更新** |

若收件箱里近期邮件已被历史运行处理过，可**再给自己发 1～2 封**不同主题的测试邮件，便于看到「新邮件进入流水线」。

### 6.2 推荐操作顺序（5～10 分钟）

1. **同时启动** 后端与前端（两个终端窗口）。
2. 浏览器打开 **http://localhost:5173** → 自动进入登录页（未登录时路由会拦截）。
3. 点击 **「使用 Microsoft 账号登录」** → 完成 Microsoft 授权 → 成功后会跳转到 **/emails**（邮件管理）。
4. 在邮件列表页点击 **拉取/同步新邮件**（具体按钮以界面为准）：
   - 默认会请求 **最近 10 封** 收件箱邮件（见 `POST /api/fetch` 的 `top` 参数；可在前端或 API 中把 `top` 调大，例如 30）。
   - 处理完成后，界面或接口返回中会有**处理数量、各状态统计、每封结果**等，可对着说明「去重、分类、自动发或进待审」。
5. 在列表中展示：
   - **状态**：待审核、已自动发送、已拒绝、发送失败、已忽略等
   - **类别**、**置信度**、**搜索/筛选**
6. 点开一封 **待审核** 或 **发送失败** 的邮件进入详情：查看**分类理由、回复草稿**；可编辑后 **批准并发送** 或 **拒绝**。（**发送失败** 状态在业务上支持再次**批准重试**发送。）
7. 打开 **数据看板**（侧栏/顶栏 **「数据看板」**）展示汇总、近 7 天趋势、类别分布等。

若演示中「全部自动发完、没有待审」：可说明「当前这封模型置信度与质量校验都通过则自动发」；或现场调高 `backend/config/thresholds.yaml` 中相关阈值后重启，再拉新邮件，更容易出现 `pending_review`。

## 7. 功能使用说明

### 7.1 拉取新邮件

调用 `POST /api/fetch`（前端按钮封装同一接口）。主要行为：

1. 从 Graph 拉取邮箱中最近邮件，数量由请求体 **`top` 控制，默认 10**（非旧文档中的 50）。
2. 按 **`message_id` 与本地库去重**，已存在则跳过。
3. **分类管线**：`categories.yaml` 中关键词/规则、业务 Gate、类别分类（含可解释 rubric 等，失败时有降级逻辑）。
4. **非业务**（`non_business` 或业务 Gate 认为非业务）：生成**说明类模板**草稿，状态 **`ignored_no_reply`**，**不自动发信**。
5. **业务类**：用**对应模板**或 LLM 生成回复草稿；随后综合**置信度**、**Rubric 是否建议自动发送**、**回复质量校验** 决定是否 `auto_sent`，否则 `pending_review`；自动发信失败记为 `send_failed` 并带重试信息。

### 7.2 邮件列表

- 状态、类别、关键词搜索、分页
- 点击行进入详情

### 7.3 邮件详情与审核

- 查看正文、分类与置信度、AI/模板生成的草稿并可编辑
- **批准并发送** / **拒绝**；批准时可提交自定义 `reply_text` 覆盖草稿（与 API 一致）

### 7.4 数据看板

- 总量、各状态、自动处理率、平均置信度、类别分布、近 7 天趋势等

### 7.5 公司产品信息

Web 端**无独立「产品管理」页面**。可任选：

- 直接维护 **`backend/data/company_products.json`**
- 或登录后通过 **PUT/POST/PATCH/DELETE** ` /api/company/products` 系列接口维护（需自行使用 API 工具）

询价类（`pricing_inquiry`）模板会**匹配主题/正文中产品名**；无匹配时可能用目录中**第一个产品**作展示（实现见 `ReplyService`）。

## 8. 可调参数与配置（YAML + 环境变量）

- **`backend/config/categories.yaml`**：类别、中英文标签、每类关键词、业务/非业务提示词
- **`backend/config/thresholds.yaml`**：全局置信度、自动发送最低置信、业务 Gate、各类路由说明与阈值（**修改后需重启后端**）
- **`backend/config/`** 下 `rubrics.yaml`、`policies.yaml` 等：评分与策略细节，见仓库内 `docs/CONFIGURATION.md`
- **`.env`**：`CONFIDENCE_THRESHOLD`、`OPENAI_*`、发信重试等

自动发送的门槛**不是**只看 `CONFIDENCE_THRESHOLD`：代码中会同时要求不低于 **`auto_send_minimum_confidence`（见 thresholds.yaml 默认 0.80）**，并受 **Rubric 与质量校验** 影响（失败会回落为待审，并在控制台/日志中可见提示）。

## 9. API 接口说明

在已登录（Session 含有效 `access_token`）前提下调用。认证相关：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/auth/login` | 跳转 Microsoft 登录 |
| GET | `/auth/callback` | OAuth 回调，写 Session 后重定向前端 `http://localhost:5173/emails` |
| GET | `/auth/logout` | 登出并清 Session |
| GET | `/auth/me` | 当前用户 |
| GET | `/auth/status` | 是否已登录（不跳转） |

邮件与统计：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/fetch` | 拉取并处理新邮件。JSON 可选 `{"top": 10}`，控制最大封数 |
| GET | `/api/emails` | 列表，支持 `status`、`category`、`search`、`page`、`per_page` |
| GET | `/api/emails/<id>` | 详情 |
| POST | `/api/emails/<id>/approve` | 审核发信。JSON 可选 `{"reply_text": "..."}` |
| POST | `/api/emails/<id>/reject` | 拒绝 |
| GET | `/api/stats` | 看板统计（含 `non_business_rate`、`send_failure_rate` 等） |

公司库：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET / PUT / POST | `/api/company/products` | 查询 / 全量替换 / 新增 |
| PATCH / DELETE | `/api/company/products/<product_name>` | 按名称更新或删除 |

`GET /api/emails` 的 `status` 取值与界面一致，例如：`pending_review`、`auto_sent`、`approved`、`rejected`、`send_failed`、`ignored_no_reply`。

## 10. 数据库与数据文件

- **`backend/email_system.db`**：主库，首次运行自动建表；运行中会按需 **ALTER** 增加 rubric、校验、PII 等列（见 `models/database.py`）
- **`backend/data/company_products.json`**：产品目录

`emails` 表除基础字段外，可包含 **`classification_rubric_scores`、`auto_send_rubric_scores`、`consent_status`、`pii_detected`、`pii_types`** 等；`replies` 可含 **校验相关 JSON 字段**。

## 11. 常见问题排查

**登录后仍回到登录页**  
检查 `AZURE_REDIRECT_URI` 与 Azure 门户注册是否完全一致；清除本站点后重试；确认前后端域名为 `localhost` 且端口与文档一致。

**AADSTS700016 等客户端 ID 错误**  
核对 `AZURE_CLIENT_ID` 与门户应用是否一致，应用是否被删除或禁用。

**拉取报 401**  
Session 过期则重新从 `/auth/login` 登录；确认权限包含 `Mail.Read` / `Mail.ReadWrite` / `Mail.Send`。

**分类或 OpenAI 报错**  
检查 `OPENAI_API_KEY`、网络、以及 `OPENAI_BASE_URL` 是否可达；查看后端控制台日志。

**页面空白或接口失败**  
确认 **5005** 与 **5173** 均在运行；浏览器开发者工具中查看跨域、Cookie 是否被拦截。

**想少自动发、多待审**  
提高 `thresholds.yaml` 中 `global` 与各类 `auto_send_threshold`，或提高 `auto_send_minimum_confidence`；重启后端后对新拉取的邮件生效。

**演示时没有「新邮件」**  
只要历史上已用同一 `message_id` 入库，再次拉取会 **skipped**。请往收件箱 **新写一封邮件** 再点拉取；或使用新的测试账号/邮箱。
