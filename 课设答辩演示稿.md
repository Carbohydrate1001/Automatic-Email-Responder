## 第一部分：系统背景与目标（30 秒）

### 问题场景
物流/贸易公司每天收到大量客户邮件：
- 询价、订单追踪、运输异常、账单查询等
- 人工处理效率低、响应慢、易出错
- 重复性工作占用大量人力

### 解决方案
**AI 驱动的自动邮件分类与回复系统**
- 自动识别邮件意图（7 大类别）
- 智能生成回复草稿
- 高置信度邮件自动发送，低置信度转人工审核

**核心价值**：提升响应速度 + 降低人力成本 + 保证回复质量

---

## 第二部分：系统架构（30 秒）

### 技术栈
```
前端：Vue 3 + Element Plus + TypeScript
后端：Flask + SQLite
AI 能力：OpenAI GPT-4o-mini
邮件服务：Microsoft Graph API (Outlook)
```

### 核心流程
```
Outlook 邮箱
    ↓ 拉取邮件
三阶段分类（规则兜底 → 业务 Gate → Few-Shot 分类）
    ↓
生成回复草稿（基于上下文提取 + 双语支持）
    ↓
置信度判断 → 自动发送 / 人工审核
    ↓
反馈记录 → 持续改进
```

## 第四部分：技术亮点与优化成果（60 秒）

### 亮点 1：分类准确性提升
**优化前问题**：
- 边界场景误判率高（如"运费多少"混淆询价和运输时间）
- 仅靠 Prompt 约束，缺乏示例指导

**优化方案**：
- ✅ Few-Shot 示例库（21 条标注样本：7 类别各 2 正例 + 1 反例）
- ✅ 7 条消歧规则（明确区分易混淆场景）
- ✅ 业务 Gate 阈值提升（0.6 → 0.7）

**效果**：分类准确性显著提升，误判率下降

### 亮点 2：回复内容恰当性
**优化前问题**：
- 回复中使用随机假数据（订单号、硬编码日期 `2026/3/16`、随机金额）
- 无语言检测，中文邮件可能收到英文回复

**优化方案**：
- ✅ 从邮件中提取真实信息（订单号、运单号、金额）
- ✅ 动态 ETA 计算（基于产品数据库的交付时长）
- ✅ 双语支持（自动检测邮件语言，生成对应语言回复）
- ✅ 产品匹配增强（支持别名和分词模糊匹配）

**效果**：回复内容真实可信，客户体验大幅提升

### 亮点 3：人工反馈回路
**创新点**：
- 所有人工操作（批准/拒绝/编辑）记录到 `feedback_records` 表
- 拒绝时可填写正确分类，用于分析改进方向
- 提供 `GET /api/feedback/stats` 接口，查询分类纠正统计和回复编辑率

**价值**：系统可持续迭代改进，越用越准

### 亮点 4：系统稳定性
- ✅ 指数退避重试（1s → 2s → 4s，上限 30s）
- ✅ 并发保护（单次拉取上限 20 封，防止 OpenAI 超限）
- ✅ 前端错误处理完善（所有异步操作均有 try-catch-finally）

## 答辩可能的提问及回答准备

### Q1：为什么选择 GPT-4o-mini 而不是其他模型？
**回答**：
- 成本考虑：mini 版本性价比高，适合高频调用
- 性能足够：邮件分类和回复生成不需要最强模型
- 响应速度快：mini 版本延迟更低，用户体验更好

### Q2：如何保证分类准确性？
**回答**：
- 三阶段分类：规则兜底 + 业务 Gate + Few-Shot 分类
- 21 条标注样本 + 7 条消歧规则
- 人工反馈回路：记录所有纠正，持续改进
- 置信度阈值控制：低置信度转人工审核

### Q3：如果 OpenAI API 不可用怎么办？
**回答**：
- 当前设计：API 失败时邮件状态标记为 `processing_failed`
- 改进方向：可增加本地模型备份（如 LLaMA）或队列重试机制

### Q4：系统安全性如何保证？
**回答**：
- OAuth2 认证：使用 Microsoft 官方认证流程
- 环境变量隔离：敏感信息（API Key、密钥）不提交代码仓库
- 前端 XSS 防护：使用 `div.textContent` 安全转义 HTML
- 后端参数校验：所有 API 输入均有类型检查

### Q5：数据库为什么选择 SQLite？
**回答**：
- 轻量级：无需独立数据库服务，适合课设演示
- 零配置：开箱即用，降低部署复杂度
- 功能完整：支持事务、索引、外键等核心功能
- 生产环境可平滑迁移到 PostgreSQL/MySQL
