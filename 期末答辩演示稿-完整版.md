## 第一部分：系统快速介绍（60 秒）

### 项目背景（15 秒）
物流/贸易公司每天收到大量客户邮件：询价、订单追踪、运输异常、账单查询等。人工处理效率低、响应慢。

**解决方案**：AI 驱动的自动邮件分类与回复系统。

### 系统架构（20 秒）
```
前端：Vue 3 + Element Plus + TypeScript
后端：Flask + SQLite  
AI：OpenAI GPT-4o-mini
邮件：Microsoft Graph API (Outlook)
```

**核心流程**：
```
拉取邮件 → AI 分类（7 大类别）→ 生成回复 → 置信度判断 → 自动发送/人工审核
```

### 功能全景（25 秒）

**7 大邮件类别**：
- 询价/报价、取消订单/退款、订单追踪
- 运输时间、运输异常、账单/发票、非业务邮件

**核心功能**：
1. **邮件列表**：状态筛选、类别筛选、关键词搜索
2. **邮件详情**：查看原始邮件、AI 分类结果、回复草稿编辑
3. **产品管理**：维护产品目录（名称、单价、MOQ、交付周期）
4. **数据看板**：自动处理率、状态分布、7 天趋势

## 第二部分：期中问题与本次优化（120 秒）

### 问题 1：分类准确性不足（40 秒）

#### 期中版本的问题
**测试案例**：
```
邮件："请问从深圳到新加坡大概要几天？"
期中分类：order_tracking（订单追踪）❌  置信度：0.82
```

**为什么错？**  
客户问的是"几天"（运输时长），应该是 `shipping_time`。但系统仅靠 Prompt 约束，无法准确区分边界场景。

#### 本次优化方案
✅ **Few-Shot 示例库**：21 条标注样本（7 类别各 2 正例 + 1 反例）
```json
{
  "category": "shipping_time",
  "label": "NEGATIVE",
  "subject": "几天能到",
  "body": "你好，我刚下单，从深圳到新加坡大概要几天？",
  "note": "反例：问的是时长而非当前状态，应归 shipping_time"
}
```

✅ **7 条消歧规则**：明确区分易混淆场景
```
1. "运费多少" → pricing_inquiry (问价格，不是时间)
2. "几天到" → shipping_time (问时长，不是查订单)
3. "我的货到哪了" → order_tracking (查当前状态)
...
```

✅ **Gate 阈值提升**：0.6 → 0.7，减少误判

#### 优化效果
```
同样的邮件："请问从深圳到新加坡大概要几天？"
优化后分类：shipping_time ✅  置信度：0.91
```

**测试数据**：7 封演示邮件，准确率从 71% → 100%

### 问题 3：无法持续改进（40 秒）

#### 期中版本的问题
**场景**：
1. 系统误判了一封邮件
2. 人工拒绝
3. **然后呢？** → 什么都没发生，系统下次还会犯同样的错误

#### 本次优化方案
✅ **新增 `feedback_records` 表**：记录所有人工操作
```sql
CREATE TABLE feedback_records (
    email_id              INTEGER,
    action                TEXT,  -- 'approved' / 'rejected' / 'approved_with_edit'
    original_category     TEXT,  -- 纠正前的分类
    corrected_category    TEXT,  -- 纠正后的正确分类
    original_reply_text   TEXT,  -- 编辑前的回复
    edited_reply_text     TEXT,  -- 编辑后的回复
    operator              TEXT,
    created_at            TEXT
);
```

✅ **拒绝时可填写正确分类**：
```typescript

```

```json
  ],
  "reply_edit_rate": 23.5  ← 23.5% 的回复被人工修改

```

## 答辩提问准备

### Q1：为什么选择 Few-Shot 而不是微调模型？
**A**：Few-Shot 更灵活，可以随时添加示例无需重新训练。微调需要大量标注数据和计算资源，不适合快速迭代。而且 Few-Shot 的效果已经足够好，测试准确率达到 100%。

### Q2：如果邮件中没有订单号怎么办？
**A**：优化后的系统会引导客户提供："请提供您的订单号或运单号，我们将立即为您跟进。"而不是像期中版本那样返回假数据。这样既保证了回复的真实性，也引导客户提供必要信息。

### Q3：反馈数据如何用于改进系统？
**A**：通过 `GET /api/feedback/stats` 查询哪些类别最常被纠正，比如发现 order_tracking 经常被纠正为 shipping_time，就针对性地补充 Few-Shot 示例或调整消歧规则。这是一个数据驱动的持续优化过程。

### Q4：系统的自动处理率是多少？
**A**：这取决于置信度阈值设置。当前设置是 0.75 且必须 >= 0.8，在测试中约 60-70% 的邮件可以自动发送。可以通过数据看板实时监控自动处理率。

### Q5：如果 OpenAI API 不可用怎么办？
**A**：当前设计是 API 失败时邮件状态标记为 `processing_failed`，可以稍后重试。未来可以增加本地模型备份或队列重试机制来提高可用性。
