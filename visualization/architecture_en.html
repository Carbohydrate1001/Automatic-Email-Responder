⚙️ System Architecture Overview

📧 Email Processing Pipeline

1. Fetch → Microsoft Graph API fetches inbox emails
2. Classify → GPT-4o-mini intent classification (rule pre-filter → business gate → category classification → rubric scoring)
3. Generate Reply → Category-based reply generation (template system + database query)
4. Score Auto-Send → Rubric scoring determines if auto-send is safe (4-dimension scoring ≥ 2.5/3.0)
5. Validate Quality → Reply quality validation (factual accuracy, tone, completeness, policy compliance)
6. Send/Review → Auto-send or route to manual review
7. Audit → Record audit log

Rubric Configuration

Classification Rubric

keyword_match 25%
intent_clarity 30%
context_completeness 25%
exclusion_confidence 20%
Formula: confidence = weighted_score / 3.0

Auto-Send Rubric

information_completeness 30%
risk_level 35%
template_applicability 20%
policy_alignment 15%
Thresholds:
≥ 2.5 auto_send_minimum
> 1 all dimensions

Reply Quality Rubric

factual_accuracy 35%
tone_appropriateness 20%
completeness 25%
policy_compliance 20%
Thresholds:
≥ 0.75 minimum_quality_score
Blocking if dimension ≤ threshold

Key Design Principles

Principle Description Implementation
Never Generate Fake Data When specific information is unavailable, acknowledge it and request clarification rather than generating fake data All templates follow: extract → validate → query → respond pattern
Request Info is Complete "Please provide missing information" is NOT a failure case - it's a complete, professional, low-risk reply Rubric scoring: information_completeness=3, risk_level=3, template_applicability=3
Database-Driven Replies Replies based on real database queries, not random generation or generic templates orders table, logistics_routes table provide real data
Explainable Confidence Use structured rubric scoring to provide explainable confidence, not black-box LLM output Each dimension 0-3 points, weighted average, with reasoning explanation
Fail-Safe Design System continues to operate safely even when API fails or data is missing Retry mechanism, circuit breaker, fallback strategy, rule pre-filter
Read-Only Reply Generation Reply generation phase should have no side effects (don't update order status, etc.) Status updates only execute after email is successfully sent
System Highlights: ① Three-layer classification mechanism reduces API call costs | ② Rubric scoring provides explainable AI decisions | ③ Database-driven approach avoids fake information | ④ Systematic handling of vague inquiries | ⑤ Multi-stage validation ensures reply quality | ⑥ Complete audit trail
