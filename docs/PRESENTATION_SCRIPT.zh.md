## 开场（约 30 秒）

> 「我们实现了一套面向**物流/贸易**场景的**自动邮件回复**系统。邮件经 **Microsoft Graph** 收进系统；**GPT** 做意图分类并起草回复；**Vue** 前端让运营人员审核与跟踪全流程。到**期中**时，我们已有：OAuth 登录、覆盖七类意图的**两阶段分类器**、高置信度**自动发送**、否则**人工审核**，以及**统计看板**。之后我们加固系统：用**多维度量表**取代原始置信度分数、在任意自动发送前增加**三阶段校验**管线、为所有外部调用加上**重试与熔断**逻辑，并建立覆盖单元、集成与端到端的**测试**套件。下面我会讲：创新点在哪里、管线**实际**如何工作、我们怎么一步步做出来，以及我们遇到的**真实**技术问题。」

## 1. 项目的创新性（约 2 分 30 秒）

先点明**业务约束**如何塑造每一个技术决策，再具体说明本系统**相对通用聊天机器人**多做了什么。

### 为什么约束重要

在物流与贸易里，**错答**不只是「没用」——可能构成**未经授权的财务承诺**或**合规风险**。所以设计问题从来不是「能不能自动化？」而是「在**什么条件下**、不经人工就发送是**安全**的？」这一约束驱动管线上的每一道**闸门**。

### 期中已具备什么

通过 Microsoft Graph 对接**真实** Outlook 邮箱。两阶段分类器：先用**规则**快速挡掉非业务邮件（无 API 代价），再对七类意图之一调 LLM（询价、查单、取消、产品咨询、投诉、退款、其他）。`.env` 里设**置信度阈值**，路由到 `auto_sent` 或 `pending_review`。发送失败有**重试**字段。SQLite 里有 `audit_log` 表。运营端有列表筛选和基础**统计**面板。这已经是**可工作的联调系统**——不是纸面原型。

### 我们补了什么、为何是「对着代码说」、而不只是「口号」

期中「是否自动发」只依赖**一个数**：`confidence ≥ threshold`。漏洞在于：LLM 可能对**承诺免运费、报未核价**的回复仍给出 `confidence=0.87`。我们用三处**落地**改法填上这个洞：

- **管线两侧都接量表评分** — `score_classification` 从四个维度（关键词与类别匹配、意图清晰度、上下文完整性、**排除**其他类的把握）计分，可用加权结果**替换**原始 LLM 置信度。`score_auto_send_readiness` 在发送前评估回复；仅**风险**一项（占比 35%）即可在综合分再高时也**否决**自动发送。两项结果都写入 **JSON 列**，使每次路由**可审计**。
- **任意自动发之前的三阶段校验** — `validate_reply_quality` 先跑**政策合规**（快、规则为主），再**幻觉/事实**检测（与公司与产品数据**对账**），最后 **LLM 质量**关。任一步未通过，代码里**关闭**自动发送，路由到 `pending_review`。这不是写进 PPT 的「政策更好」，而是 `process_email` 里的**硬分支**。
- **任意 LLM 调用前先扫 PII** — `detect_pii` 在进分类前跑在主题与正本上；检出的类型带**脱敏标记**入库，**不以明文**进日志。

**一句收束**：「创新点在于：『自动发送』**不是**一个置信度门槛——而是**五道顺序闸门**，**任何一道**都能拦住这封回复。」

## 2. 技术复杂度（约 2 分钟）

复杂度**不是**服务数量，而是**两个方法内部**被**强制的**操作顺序，以及任一步失败时发生什么。

### 入口

`POST /api/fetch` 从 Graph 拉信、按 `message_id` 在 SQLite 里**去重**；对每条新信依次调用 `classify_email` 再 `process_email`。流经系统的每封邮件都会打一条带 `status`、`category`、`confidence`、`latency_ms` 的 `[PIPELINE]` 日志。对外集成面就这些——**一个路由、每封邮件两个方法**。

### `classify_email` 内部——五步**固定**顺序

一、**关键词/规则**快路径：明显非业务邮件、**不**调 API。二、**LLM 业务门**：不过的直接标 `non_business` 早退。三、JSON 模式 OpenAI 调参返回 `category` 和原始 `confidence`（正文截断 3000 字，置信度钳到 `[0,1]`）。四、`score_classification` 按四维权重量表计分；综合分过线则**替换**结果里的原始置信度。五、LLM 调用包在 `@with_retry` 和**熔断**里；若重试**全部**失败，`_fallback_classification` 返回关键词匹配结果、置信度上限 0.7、记录里打 `fallback_used=true`。

### `process_email` 内部——七步**固定**顺序

对进线主题/正文做 **PII 扫描**。**生成回复**（模板 + GPT 兜底，产品数据来自 `company_products.json`）。按 `.env` 阈值做**是否可自动发**的初判。`score_auto_send_readiness` 量表计分：四维——信息完整度（0.30）、**风险**（0.35）、模板适用性（0.20）、与政策一致（0.15）；**风险分为 0 即硬否决**。若仍具备资格：**阶段一** `check_policy_compliance` 用 `policies.yaml` 禁则 **regex** 拦；**阶段二** `detect_hallucinations` 从回复里抽**价格、产品说法、时间承诺**并与公司数据文件**逐条对账**，任一条 `false` 即拦；**阶段三** LLM 评语气、完整度与事实性。任阶段 `passed=false` 会清掉 `auto_send_eligible`，邮件进 `pending_review`。**仅当七步都过**才调用 `graph_service.send_reply`。量表分、校验结果、PII 标记**每封**都进 SQLite 的 JSON 列。

### 可配置面

`rubrics.yaml`：维度权重、每维 0–3 分刻度、**综合**阈值。`policies.yaml`：禁则（如「保证」、具体金额、「不关我们事」等）与**预批准**措辞列表。`categories.yaml`：七类意图。调自动发送敏感度、或**封禁**新说法——**改文件**即可，**不必**改业务代码。

**可选 10 秒收束**：「复杂度 = **一个**路由、**两个**方法里**强制**的七步顺序、**每个**决策**写进** SQLite，所以你能**始终说清**某封是发了还是**被扣下**。」

## 3. 实施过程（约 2 分钟）

诚实的叙述是**顺序**：在能用测试证明对 `classify_email`、`process_email` 的修改**不会**把既有路由打烂之前，我们**不能**安全地堆评分、校验与弹性。因此顺序是**刻意**的。

### 第一步：能罩住**期中行为**的测试网

在动任何业务逻辑前，我们搭了 **pytest**：内存 **SQLite**、**一致**的 OpenAI 与 Graph **mock**、七类意图的中英 **fixture** 邮件。首版用例覆盖：含非业务早退的**七条**分类路径；置信度钳位与边界（空正文、3000 字截断）；不同阈值组合下自动发 vs 待审；发送失败与 `send_failed` 分支；各分类**模板**选择；以及 Unicode-only、无主题等**畸形**输入。这让我们在后续改动前就有了**回归网**。

### 第二步：用多维度评分取代**单一**置信度

测试就位后，在现有方法里加入 `score_classification` 与 `score_auto_send_readiness`。分类量表看：关键词是否与所选类匹配、意图是否清楚、上下文是否够、其他类能否有把握排除。自动发量表看：是否**信息**足够写出完整回复、**回复内容**风险、标准模板**贴合度**、**政策**一致。**风险**维权最重（35%），因为「分类分低/风险分低但分类**很自信**」仍可能产出一笔**未授权**承诺。随后做**校准**轮：看原始 LLM `confidence` 是否**真**校准、对明显偏的类**单独**设阈值（写在 `rubrics.yaml` 等）。

### 第三步：用回复校验补量表**够不着**的

量表在**未生成**回复前评「就绪度」；**校验**盯的是**生成后的**文字。三阶段**先便宜后贵**：**regex** 政策在毫秒级拦明显违规；**抽主张 + 对账**拦编造价格/规格；**LLM 质量**拦语气差或答非所问。在接入 `process_email` 前，我们**分阶段**单测：全过、**单维**挂、**多维**问题、空回复与 HTML 等边界。

### 第四步：包紧每一次外部调用

核心管线稳定并测通后，为所有 OpenAI 与 Graph 调用加**指数退避**（基准 1 秒、每次 ×2、±10% 抖动、上限 60 秒）与**熔断**：OpenAI 分类**连续 5 次**失败则开路、60 秒后再试；Graph 更**紧**（3 次、120 秒）——**发送**失败直接影响客户。状态可在 `GET /health/circuit-breakers` 查。用专门用例验重试/兜底：首次成功、**两次**后成功、**三次**耗尽、**熔断开路**时关键词分类兜底是否**正确**。

### 第五步：PII 扫描、语言检测与剩余覆盖

管线入口加 **PII**（7 种 regex：电话、邮箱、地址、证件号、卡号、人名模式、银行账号等）。**语言**用 **Unicode 区块比例**（CJK vs 拉丁）选中英模板、**不**再依赖额外线程库。端到端在 mock 外部依赖下跑通 `fetch → classify → score → validate → route`；**回归**用一组「黄金路径」邮件**锁死**预期路由。

**一句收束**：「每一步都多了一条**可验证**的保证；到最后，**仅当**分类置信度、量表分与**三道**校验**一致**同意，才会真的自动发。」

## 4. 主要挑战与应对（约 2 分 30 秒）

| 挑战 | 具体缺口 | 我们做的 | 可观察结果 |
| --- | --- | --- | --- |
| **LLM 置信度未校准** | `confidence=0.87` 不等于 87% 对；门槛曾是「拍脑袋」 | 四维分类量表；Platt / 等渗回归**校准**分析；`rubrics.yaml` **按类**阈值 | 自动发门槛**基于测量校准**，不是猜 |
| **分类很自信，回复仍可能很糟** | 高置信度 → 自动发，**没**审**生成内容**本身 | `score_auto_send_readiness` 风险维**硬否决**；`validate_reply_quality` **三阶段**管线 | **未授权承诺、未证实说法**的回复在发出前被拦 |
| **政策违规对 LLM 不可见** | LLM 不知道公司**能/不能**承诺什么 | `check_policy_compliance` 对 `policies.yaml` 做 **regex**；禁则含具体报价、「保证」、全额退款、甩锅话术等 | 政策问题在**阶段一**被拦，**早于**更慢的 LLM 校验 |
| **LLM 会编造产品细节** | 如「三天到货」「$120」可能瞎编 | `detect_hallucinations` 抽事实主张（价、规格、时间）并对 `company_products.json` 核实；`false` 则关自动发 | **事实编造**进**人工** |
| **Graph / OpenAI 间歇失败** | 期中仅在发送侧有简单重试字段 | 全 LLM 调用 `@with_retry` + 指数退避 + ±10% 抖动；OpenAI 与 Graph **分设**熔断（5 次 / 3 次）；熔断开路时**关键词**分类兜底 | 短期 API 故障时系统**仍能扛**；运营可看 `/health/circuit-breakers` |
| **改路由可能悄悄**弄坏**旧行为** | 初版测试面小、缺边界与**全链** | **200+** 用例，单元/集成/边界/端到端；**黄金路径**回归；**显式**测兜底、PII、语言 | 之后对 `classify_email` / `process_email` 的改动**合入前**都跑**全套** |

**收束（约 20 秒）**：「这些问题的**根**都一样：**LLM 输出是概率的，但业务邮件承诺不是。**工程答案是在**概率核心**外叠**确定性的**层——量表、政策 regex、事实对账、熔断——这样**仅当**系统**能为**这次自动发**辩护**时，它才会发。」

## 5. Q&A 预备

- **对比通用聊天机器人**：接**真实** Outlook 邮箱、**固定**七类体系、未达门槛进**人工队列**、LLM **绕不过**的硬政策与 PII 层。
- **稳定性怎么说**：重试 + 熔断 + 200+ 测。**不要**吹生产 SLA——这是课程项目、实验环境、接**真实** API。
- **隐私**：PII 与审计**已实现**并测过；若要做**全公司**上线，还要正式的数据分级与**留存**评审，**超出**课程范围。
- **为什么不把置信度阈值**再**调高**？：阈值调高会减少自动发**量**，但**治不了**校准——模型**错得照样自信**。量表与校验管的是**决策质量**，不仅是**自动发比例**。
