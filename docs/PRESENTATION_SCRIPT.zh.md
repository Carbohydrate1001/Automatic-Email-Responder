## 主线（可选幻灯，约 15 秒说清）

1. **问题**：物流/贸易场景下客户邮件量大、问题重复；需要自动化，但不能**盲目**自动发出。  
2. **稳定性**：针对外部 API 不稳定与 LLM 输出方差，用**重试、熔断、健康检查、测试**加固整条管线。

## 开场（约 30 秒）— **中文口播稿**

> 「我们实现了一套面向**物流/贸易**场景的**客户邮件自动回复**系统。邮件经 **Microsoft Graph** 读取与发送；**GPT** 负责意图分类与回复草稿；**Vue** 前端提供审核与统计。到**期中**时，已具备：OAuth 登录、**两阶段分类器**、回复草稿、**高置信度自动发送** 或 否则**人工审核**、以及**统计看板**。之后我们**持续迭代实现**：配置外置、基于**量表**的评分与**置信度校准**、**多阶段回复校验**管线、**弹性**模式、**隐私与合规模块**、**多语言**处理，以及规模较大的**测试**套件。下面按**创新性、技术复杂度、分阶段怎么做的、主要技术挑战**四块展开。」

## 1. 项目的创新性（约 2 分 30 秒）— **中文**

**讲法建议**：先用 **2～3 句**讲清价值，再对比**通用聊天机器人**与**期中基线**，说明系统**具体不同在哪**。

**建议口播要点：**

- **干系人价值（简要）**：把标准化问询从人工里**省出来**；用**统一分类体系**和 **JSON 产品库** 保证回答与公司事实一致（[使用说明.md](../使用说明.md)）。错答或**过度承诺**会牵涉**品牌与责任** —— 所以产品不是「为回而回」，而是**在安全时再自动或放行**。  
- **期中（对课程项目已足够有新意）**：真实 **Outlook** 对接、**7 类**意图、**规则 + LLM** 门控、**非业务**通道、**待审核** 与 **已自动发送** 分流、失败时的**重试字段**、SQLite 里的 **audit_log**、**运营向**的列表筛选与统计。  
- **后续迭代（对照代码，而不是功能清单式堆砌）**：
  - **分类与自动发两侧都接量表** — `score_classification` 与 `score_auto_send_readiness` 写在现有 **`classify_email` / `process_email`** 里，会**改写**标量置信度、以及是否满足自动发（详见 **§2**）。  
  - **自动发被三阶段校验卡住** — `validate_reply_quality` 在实现里是 **政策合规 → 幻觉/事实 → LLM 质量** 顺次执行；任一步不通过，代码里会**关掉** `auto_send_eligible`，不是口头「策略更好」。  
  - **进线先做 PII 与合规模块挂钩** — 对主题/正文 `detect_pii`，再 `log_audit_event`；SQLite 增加**量表/PII 等字段**，不是另做一套「合规中台」。  
- **一句收束**：「创新点仍然是：**一条真实发件链路 + 可测的闸门**——一个路由、两个核心方法、中间每步有落库。」

## 2. 技术复杂度（约 2 分钟）— **中文，以实现为主**

**目标**：让听众觉得这是**能交付**的软件—点出**组件、配置、数据流**，每条对应**具体技术职责**。

- **身份与平台**：**MSAL**、OAuth2、带 `access_token` 的 Flask **Session** 调 Graph；开发态 CORS / Vite 代理（[使用说明.md](../使用说明.md) API 表）。难点：**令牌生命周期**、委托权限 `Mail.Read` / `Mail.ReadWrite` / `Mail.Send`。  
- **接入与去重**：**Graph** 拉取（如最近一批），SQLite 中 **`message_id` 唯一** — 避免重复处理。  
- **分类管线**（[ARCHITECTURE.md](ARCHITECTURE.md) 之 `classification_service`）：**关键词/规则**快路径、**LLM 业务门**、**LLM 细类**；置信度限幅；`config/categories.yaml` 经 `config_loader` 驱动。  
- **评分**（`scoring_service`）：`config/rubrics.yaml` 多**维度量表**、`config/thresholds.yaml` **按类阈值** — 自动发送绑定**可解释**维度，而非单标量。  
- **回复生成**（`reply_service` + `company_info_service`）：**模板** + **GPT** 回退；产品数据见 `company_products.json` 与 [使用说明.md](../使用说明.md) 中 API。  
- **生成后校验**（`validation_service` + `config/policies.yaml`）：允许自动发之前的**多阶段**检查。  
- **弹性**（`utils/retry_handler.py`）：**指数退避**、**抖动**、**熔断**，减轻 Graph/OpenAI **级联**故障。  
- **可观测与运维**：结构化**日志**（`utils/logger.py`）、`api/health.py`、`api/metrics.py`（[ARCHITECTURE.md](ARCHITECTURE.md)）。  
- **测试**：**200+** 用例，单元/集成/边界/端到端/回归（[archive/FINAL_REPORT.md](archive/FINAL_REPORT.md)）— 对 **OpenAI** 与 **Graph** 做**一致** mock 成本高。

**可选一句架构**：「**浏览器 → Flask → Graph 与 OpenAI → SQLite**；**YAML 配置**驱动类别、量表、阈值与策略；**分类** 与 **发送** 之间夹 **评分与校验**；**重试与熔断**包在外呼外侧。」

## 3. 实施过程（约 2 分钟）— **中文，按阶段技术迭代**

**讲法**：这是**工程时间线**，不是市场话术；每一阶段 = **可指认的交付物**。

- **Phase 0 / 期中**（[使用说明.md](../使用说明.md)）：端到端 **MVP** — 登录、拉信管线、两阶段分类、回复草稿、路由到 **auto_sent / pending_review / send_failed / ignored_no_reply**、基础统计、公司 JSON、SQLite 结构。  
- **Phase 1**（[archive/PHASE1_SUMMARY.md](archive/PHASE1_SUMMARY.md)）：**pytest** 框架、初期 **145+** 用例、**配置外置** — 为后续安全重构打基础。  
- **Phase 2.1–2.2**：**`scoring_service` + 量表**；**Platt / 等渗**校准、阈值调优、**CALIBRATION_REPORT** 式分析 — 替代粗放的单分数门槛。  
- **Phase 3**：**`validation_service`**、扩展的回复质量**量表**、硬失败时**拦截自动发**。  
- **Phase 4**：**`retry_handler`**、健康端点、LLM 不可用时的**降级**路径（[archive/FINAL_REPORT.md](archive/FINAL_REPORT.md) 提及关键词分类等）。  
- **Phase 5**：**`pii_service`**、**compliance_audit_log**、留存 / GDPR 向 API — 见 [ARCHITECTURE.md](ARCHITECTURE.md) 模式扩展。  
- **Phase 6**：E2E、**回归**黄金集、**A/B** 工具（`ab_testing.py`）、**metrics** API。  
- **Phase 7**：**`language_service`**、CJK 与拉丁字符比例、按语言调**模板**。  
- **Phase 8**：**文档**（ARCHITECTURE、API_REFERENCE、DEPLOYMENT、OPERATOR_MANUAL 等，见 [README.md](README.md)）。

**收束（技术向）**：「我们**自底向上**迭代：**先测试与配置**，再**评分与校验**，再**可靠性与隐私**，再**评估与多语言**，最后**打包与文档** —— 让下一层动土前，上一层已有测试网。」

## 4. 主要挑战与应对（约 2 分 30 秒）— **中文，技术优先**

**口播结构**：*工程/模型挑战* → *期中/朴素做法缺什么* → *我们怎么实现* → *（可选）对外效果*。

| 挑战 | 期中/朴素不足 | 工程应对（位置） | 简要结果 |
|------|----------------|------------------|----------|
| **原始置信度不稳** | 单一 `confidence` + 环境阈值；代码里还有例如 ≥0.8 的硬门（[使用说明.md](../使用说明.md) 注） | **量表评分**、**Platt/等渗**校准、**按类** YAML 阈值（`scoring_service`，Phase 2） | 自动发送规则**更可解释、可调** |
| **自动回复不严谨** | 生成后几乎直接发，缺深度检查 | 多阶段 `validation_service`、`policies.yaml`（Phase 3） | 明显不合格的回复**不会**被自动发出 |
| **外部 API 不稳定** | 仅发送侧环境变量**重试**等 | **指数退避+抖动+熔断**、健康检查（Phase 4，`retry_handler`） | **少堆积、少**惊群**式重试** |
| **正文与日志中的 PII** | 期中着墨少 | `pii_service` **regex** 类型、**脱敏**级别、更安全的日志（Phase 5） | 运营侧**泄露风险**降低 |
| **多语言真实邮件** | 偏英文模板 | `language_service` **Unicode** 分析；中/英 fixture（Phase 7） | **母语句式**的回复更自然 |
| **每次改动怕回归** | 初期测试面小 | **200+** 用例，e2e+回归（Phase 1、6） | 发布**更有把握** |
| **可运维性** | 排障靠临时查 | **JSON 日志**、**metrics** 端点、运维/部署**文档** | 故障**定位更快** |

**收束（约 20 秒，技术+干系人各带一句）**：「技术上最难的是把 **LLM 的软质量** 和 **工程上的硬约束** 接在一起：用**校准与量表**管**何时**自动行动，用**校验**管**发什么**，用**弹性**管 **Graph/OpenAI 出问题**时系统**怎么活**。对业务方，就是**更安全的自动化**、**更可控的可用性**。」
