## 开场（约 30 秒）

> 「我们实现了一套面向物流/贸易场景的自动邮件回复系统。邮件经 Microsoft Graph 收进系统；GPT 做意图分类并起草回复；Vue 前端让运营人员审核与跟踪全流程。到期中时，我们已有：OAuth 登录、覆盖七类意图的两阶段分类器、高置信度自动发送、否则人工审核，以及统计看板。之后我们加固系统：用多维度量表取代原始置信度分数、在任意自动发送前增加三阶段校验管线、为所有外部调用加上重试与熔断逻辑，并建立覆盖单元、集成与端到端的测试套件。下面我会讲：创新点在哪里、管线实际如何工作、我们怎么一步步做出来，以及我们遇到的真实技术问题。」

## 1. 项目的创新性（约 2 分 30 秒）

这一部分不要讲成阶段性回顾，而要直接回答一个问题：**相对于一个“会写邮件”的通用大模型系统，我们这个项目真正创新在哪里？**

### 为什么约束重要

在物流与贸易里，错答不只是「没用」——可能构成未经授权的财务承诺或合规风险。所以设计问题从来不是「能不能自动化？」而是「在什么条件下、不经人工就发送是安全的？」这一约束驱动管线上的每一道闸门。

### 创新点在哪里

本项目的创新，不在于“用了 GPT 写回复”，而在于把**自动邮件回复**做成了一套面向真实业务场景的**可控决策系统**。相对通用聊天机器人，我们的创新主要体现在三点：

- **创新一：把“生成回复”升级为“端到端闭环处理邮件”** — 系统不是停在一个聊天框里生成一段文本，而是直接接入真实 Outlook 邮箱，通过 Microsoft Graph 收信、分类、生成回复、决定自动发送还是转人工审核，并把结果回写到运营端和数据库。也就是说，我们做的不是“本地模型演示”，而是一个**能够进入真实业务流程**的邮件处理闭环。
- **创新二：把“模型觉得像”升级为“系统根据rubric给出评估”** — 通用聊天机器人通常依赖单次生成结果或单一置信度（低可信度）；我们的系统把分类判断和发送判断都拆成**多维度、带权重、可配置的量表**。例如，邮件是否适合自动发送，不只看回复像不像，还单独看**风险等级、信息完整度、模板适用性、政策一致性**。这些评分会结构化写入数据库并比对评估，因此系统不仅能做决定，还能解释“**为什么发**”或“**为什么不发**”。
- **创新三：把“能答出来”升级为“答得安全、可审计、可兜底”** — 真正难的不是让模型写一句顺畅的话，而是保证它**不会乱承诺、不会编造事实、不会在敏感信息上失控**。因此我们在自动发送前设置了固定顺序的质量闸：先做政策合规检查，再做事实/幻觉对账，最后做整体质量评估；同时对客户敏感信息做识别、脱敏和审计记录。只要任一环节不过，系统就默认不自动发，转入 `pending_review` 交给人工确认。

一句收束：「**这个项目的创新，不是让大模型“会回邮件”，而是让系统在真实业务里“知道什么时候能回、为什么能回，以及什么时候必须交给人”。**」

## 2. 技术复杂度（约 2 分钟）

复杂度不是服务数量，而是两个方法内部被强制的操作顺序，以及任一步失败时发生什么。

### 入口

`POST /api/fetch` 从 Graph 拉信、按 `message_id` 在 SQLite 里去重；对每条新信依次调用 `classify_email` 再 `process_email`。流经系统的每封邮件都会打一条带 `status`、`category`、`confidence`、`latency_ms` 的 `[PIPELINE]` 日志。对外集成面就这些——一个路由、每封邮件两个方法。

### `classify_email` 内部——五步固定顺序

一、关键词/规则快路径：明显非业务邮件、不调 API。二、LLM 业务门：不过的直接标 `non_business` 早退。三、JSON 模式 OpenAI 调参返回 `category` 和原始 `confidence`（正文截断 3000 字，置信度钳到 `[0,1]`）。四、`score_classification` 按四维权重量表计分；综合分过线则替换结果里的原始置信度。五、LLM 调用包在 `@with_retry` 和熔断里；若重试全部失败，`_fallback_classification` 返回关键词匹配结果、置信度上限 0.7、记录里打 `fallback_used=true`。

### `process_email` 内部——七步固定顺序

对进线主题/正文做 PII 扫描。生成回复（模板 + GPT 兜底，产品数据来自 `company_products.json`）。按 `.env` 阈值做是否可自动发的初判。`score_auto_send_readiness` 量表计分：四维——信息完整度（0.30）、风险（0.35）、模板适用性（0.20）、与政策一致（0.15）；风险分为 0 即硬否决。若仍具备资格：阶段一 `check_policy_compliance` 用 `policies.yaml` 禁则 regex 拦；阶段二 `detect_hallucinations` 从回复里抽价格、产品说法、时间承诺并与公司数据文件逐条对账，任一条 `false` 即拦；阶段三 LLM 评语气、完整度与事实性。任阶段 `passed=false` 会清掉 `auto_send_eligible`，邮件进 `pending_review`。仅当七步都过才调用 `graph_service.send_reply`。量表分、校验结果、PII 标记每封都进 SQLite 的 JSON 列。

### 可配置面

`rubrics.yaml`：维度权重、每维 0–3 分刻度、综合阈值。`policies.yaml`：禁则（如「保证」、具体金额、「不关我们事」等）与预批准措辞列表。`categories.yaml`：七类意图。调自动发送敏感度、或封禁新说法——改文件即可，不必改业务代码。

可选 10 秒收束：「复杂度 = 一个路由、两个方法里强制的七步顺序、每个决策写进 SQLite，所以你能始终说清某封是发了还是被扣下。」

## 3. 实施过程（约 2 分钟）

诚实的叙述是顺序：在能用测试证明对 `classify_email`、`process_email` 的修改不会把既有路由打烂之前，我们不能安全地堆评分、校验与弹性。因此顺序是刻意的。

### 第一步：能罩住期中行为的测试网

在动任何业务逻辑前，我们搭了 pytest：内存 SQLite、一致的 OpenAI 与 Graph mock、七类意图的中英 fixture 邮件。首版用例覆盖：含非业务早退的七条分类路径；置信度钳位与边界（空正文、3000 字截断）；不同阈值组合下自动发 vs 待审；发送失败与 `send_failed` 分支；各分类模板选择；以及 Unicode-only、无主题等畸形输入。这让我们在后续改动前就有了回归网。

### 第二步：用多维度评分取代单一置信度

测试就位后，在现有方法里加入 `score_classification` 与 `score_auto_send_readiness`。分类量表看：关键词是否与所选类匹配、意图是否清楚、上下文是否够、其他类能否有把握排除。自动发量表看：是否信息足够写出完整回复、回复内容风险、标准模板贴合度、政策一致。风险维权最重（35%），因为「分类分低/风险分低但分类很自信」仍可能产出一笔未授权承诺。随后做校准轮：看原始 LLM `confidence` 是否真校准、对明显偏的类单独设阈值（写在 `rubrics.yaml` 等）。

### 第三步：用回复校验补量表够不着的

量表在未生成回复前评「就绪度」；校验盯的是生成后的文字。三阶段先便宜后贵：regex 政策在毫秒级拦明显违规；抽主张 + 对账拦编造价格/规格；LLM 质量拦语气差或答非所问。在接入 `process_email` 前，我们分阶段单测：全过、单维挂、多维问题、空回复与 HTML 等边界。

### 第四步：包紧每一次外部调用

核心管线稳定并测通后，为所有 OpenAI 与 Graph 调用加指数退避（基准 1 秒、每次 ×2、±10% 抖动、上限 60 秒）与熔断：OpenAI 分类连续 5 次失败则开路、60 秒后再试；Graph 更紧（3 次、120 秒）——发送失败直接影响客户。状态可在 `GET /health/circuit-breakers` 查。用专门用例验重试/兜底：首次成功、两次后成功、三次耗尽、熔断开路时关键词分类兜底是否正确。

### 第五步：PII 扫描、语言检测与剩余覆盖

管线入口加 PII（7 种 regex：电话、邮箱、地址、证件号、卡号、人名模式、银行账号等）。语言用 Unicode 区块比例（CJK vs 拉丁）选中英模板、不再依赖额外线程库。端到端在 mock 外部依赖下跑通 `fetch → classify → score → validate → route`；回归用一组「黄金路径」邮件锁死预期路由。

一句收束：「每一步都多了一条可验证的保证；到最后，仅当分类置信度、量表分与三道校验一致同意，才会真的自动发。」

## 4. 主要挑战与应对（约 2 分 30 秒）


| 我们遇到的关键挑战 | 如何应对 |
| --- | --- |
| 置信度与真实对错脱节：例如 `confidence=0.87` 并不表示约 87% 判对；大模型直接输出的是自报分数，在未校准前不能当作与真实判对率一致的可操作概率，单拿它定自动发门槛依据缺乏可信度。 | 用四维分类量表重述「该不该信这次分类」；用 Platt / 等渗回归做校准；在 `rubrics.yaml` 里按类设阈值，使门槛由对历史表现的测量与校准决定，而非未校验的模型自报值。 |
| 只盯分类、不盯文字：高置信时仍可能生成带风险承诺或答非所问的回复，仅靠分类无法在发出前拦。 | 在生成前用 `score_auto_send_readiness`（风险维硬否决）拦「能发吗」；生成后用 `validate_reply_quality` 三阶段管线审字面上的合规与质量。 |
| 政策是「公司规则」、不是模型常识：LLM 不知道能承诺什么/禁说什么，会写出看似合理的违规话。 | 用 `check_policy_compliance` 对 `policies.yaml` 做regex 快速扫禁则（如具体报价、「保证」、全额退款、推诿或转嫁责任等表述），在最便宜的阶段一就拦，先于更慢的 LLM 校验。 |
| 产品事实可能编造：例如「三天到货」「$120」可来自幻觉，对不上真实目录与价格。 | `detect_hallucinations` 抽价、规格、时间等事实主张，逐条对 `company_products.json` ；对不上则关自动发、转人工。 |
| 外部 API 不稳定：OpenAI 与 Microsoft Graph 会间歇失败；期中仅在发送路径配置重试，未为分类与多轮 LLM 调用提供对等的退避与容错机制。 | 全 LLM 相关调用统一 `@with_retry`（指数退避 + ±10% 抖动）；OpenAI 与 Graph 分设熔断（5 次 / 3 次）；熔断开路时关键词分类兜底；状态可查 `/health/circuit-breakers`。 |
| 动 `classify_email` / `process_email` 时，容易「改坏旧行为」：动一行路由或一步校验，可能让过去处理对的邮件现在进错状态，而手工很难在每次小改后都逐封复查。早期若只测主路径、不测边界和整条 fetch→发信链路，这类问题更容易漏到上线后。 | 用 200+ 条自动用例，把从典型邮件到最终路由（待审/已发等）的期望结果固定住（单元、集成、边界、E2E），并设「黄金邮件」基线；再单独为分类兜底、PII、语言检测写用例。之后每次改这两处函数，合入前在 CI 必须跑全量用例，不通过就不合并，避免旧行为被改坏却未被发现就合进主干。 |

收束（约 20 秒）：「挑战可以概括成：模型给的是概率输出，可业务邮件要的是可辩护的承诺。我们的应对是在概率核心外面加一层层可执行、可复测的规则——量表与校准、政策 regex、事实对账、重试与熔断、充分的测试覆盖——仅当这些规则在决策上对本次自动发同时成立时，系统才自动发送。」

## 5. Q&A 预备

- 对比通用聊天机器人：接真实 Outlook 邮箱、固定七类体系、未达门槛进人工队列、LLM 绕不过的硬政策与 PII 层。
- 稳定性怎么说：重试 + 熔断 + 200+ 测。不要吹生产 SLA——这是课程项目、实验环境、接真实 API。
- 隐私：PII 与审计已实现并测过；若要做全公司上线，还要正式的数据分级与留存评审，超出课程范围。
- 为什么不把置信度阈值再调高？：阈值调高会减少自动发量，但治不了校准——模型错得照样自信。量表与校验管的是决策质量，不仅是自动发比例。
